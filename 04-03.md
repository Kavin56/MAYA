- **Browser console errors (root cause)**
  - When the UI tries to talk directly to the ngrok-hosted OpenWork server, the browser blocks the calls because there is no CORS header on the responses:
    ```text
    Access to fetch at 'https://nondetonating-cecile-nongrounded.ngrok-free.dev/health'
    from origin 'https://tubular-nasturtium-0e2576.netlify.app' has been blocked by CORS policy:
    Response to preflight request doesn't pass access control check:
    No 'Access-Control-Allow-Origin' header is present on the requested resource.
    ```
  - Similar errors appear for:
    - `https://nondetonating-cecile-nongrounded.ngrok-free.dev/token`
    - `https://nondetonating-cecile-nongrounded.ngrok-free.dev/opencode/global/health`
  - As a result, the browser never sees a successful `/health` or `/status` response, and the UI keeps the status as `MAYA Unavailable` and `OpenCode Not connected`.

### Summary of the blocking issue

- The **backend is running correctly** (OpenCode + maya-server + ngrok public URL + `/token`).
- The **frontend is configured to target the correct URL** via `VITE_OPENWORK_URL` and tries to fetch `/token`, `/health`, and `/opencode/global/health`.
- The **browser blocks these requests due to CORS**, because the OpenWork server responses (including `/health` and `/token`) do not include an `Access-Control-Allow-Origin` header that allows `https://tubular-nasturtium-0e2576.netlify.app` (or `*`).
- Until CORS headers are correctly returned by the OpenWork server for these routes, the hosted web UI cannot mark the server as connected, even though the server itself is healthy and reachable.
